INTERMEDIATE CODE IN THE URIUM COMPILER

 

The URIUM compiler uses its own intermediate code consisting of a total of 24 instructions. It's a three-adress code where addresses are stored in the target, source1, and source2 fields. The instructions used are as follows:

  • LABEL

Describes a label that can be jumped to. The target field stores the label's identifier.

  • ASSIGN

Describes an assignment statement. The target field stores a reference to the target variable, and the source1 field stores the source of the assignment, which can be a reference to a variable or a constant value (a literal).

  • ADD

Describes an addition instruction. The target field stores the reference to the destination variable, and the source1 and source2 fields store the operands.

  • SUB

Describes a subtraction instruction. The target field stores the reference to the destination variable, and the source1 and source2 fields store the operands.

  • MUL

Describes a multiplication instruction. The target field stores the reference to the destination variable, and the source1 and source2 fields store the operands.

  • DIV

Describes a division instruction. The target field stores the reference to the destination variable, and the source1 and source2 fields store the operands.

  • MOD

Describes an instruction that computes the remainder of an integer division. The target field stores the reference to the destination variable, and the source1 and source2 fields store the operands.

  • INV

Describes a sign change instruction. The target field stores the reference to the target variable, and the source1 field stores the operand.

  • AND

Describes a logical conjunction (and) instruction. The target field stores the reference to the destination variable, and the source1 and source2 fields store the operands.

  • OR

Describes a logical disjunction (or) instruction. The target field stores the reference to the destination variable, and the source1 and source2 fields store the operands.

  • NOT

Describes a logical negation (not) instruction. The target field stores the reference to the destination variable, and the source1 field stores the operand.

  • JUMP

Describes an unconditional jump instruction. The target field stores the jump label identifier.

  • JMPEQ

Describes a jump instruction conditional on equality between the operands. The target field stores the jump label. The source1 and source2 fields contain references to the data to be compared. If source1 is equal to source2, the jump occurs.

  • JMPNE

Describes a jump instruction conditional on unequality between the operands. The target field stores the jump label. The source1 and source2 fields contain references to the data to be compared. If source1 is different from source2, the jump occurs.

  • JMPGT

Describes a jump instruction conditional on a "greater than" comparison. The target field stores the jump label. The source1 and source2 fields contain references to the data to be compared. If source1 is greater than source2, the jump occurs.

  • JMPGE

Describes a jump instruction conditional on a "greater than or equal" comparison. The target field stores the jump label. The source1 and source2 fields contain references to the data to be compared. If source1 is greater than or equal to source2, the jump occurs.

  • JMPLT

Describes a jump instruction conditional on a "less than" comparison. The target field stores the jump label. The source1 and source2 fields contain references to the data to be compared. If source1 is less than source2, the jump occurs.

  • JMPLE

Describes a jump instruction conditional on a "less than or equal" comparison. The target field stores the jump label. The source1 and source2 fields contain references to the data to be compared. If source1 is less than or equal to source2, the jump occurs.

  • JMP1

Describes a jump instruction conditioned on a Boolean value. The target field stores the jump label. The source1 field contains the reference to the Boolean data. If source1 is 1 (encoded as true), the jump will occur.

  • PARAM

Describes an instruction for storing a procedure call parameter. Before calling the procedure, the parameters must be evaluated and their values stored. The target field contains the value to be stored. The source1 and source2 fields contains the parameter position in the procedure call.

  • CALL

Describes a procedure call instruction. The target field contains the procedure identifier (the procedure's start label). The call parameters must be stored in advance using PARAM instructions.

  • ENDP

Describes the termination instruction for a procedure. It does not require the use of any fields.

  • PRECALL

Describes a procedure call preparation instruction. The target field contains the number of basic type parameters the call will have. The source1 field contains the number of reference type parameters the call will have. The procedure call process begins by indicating the number of parameters for the call (PRECALL), then the parameter values are calculated and stored in their location (PARAM), and finally the call is made (CALL).

  • POINTER

Describes an instruction that obtains a reference (the address) to a variable. The target field contains the destination variable. The source1 field contains the variable to be referenced.

 

 

CLASSES USED TO REPRESENT INTERMEDIATE CODE

 

The urium.code package contains the following classes dedicated to representing instructions in intermediate code:

  • urium.code.CodeConstants

This is an interface that defines the constants that identify the types of instructions in the intermediate code.

  • urium.code.CodeAddress

Abstract class that describes an address in three-address code. Addresses used in instructions can be labels, literals (constant values), and variable references.

  • urium.code.CodeLabel

Subclass of CodeAddress. Defines a label that can be used as a field in intermediate code instructions.

  • urium.code.CodeLiteral

Subclass of CodeAddress. Defines a constant value (a literal) that can be used as a field in intermediate code instructions. The constant value is stored in hexadecimal format.

  • urium.code.CodeVariable

Subclass of CodeAddress. Defines a reference to a variable that can be used as a field in intermediate code instructions. Intermediate code generation transforms variable names to avoid duplication. The original name of the variable (initially described in a urium.ast.struct.Variable object) is stored in the des field. The new name is stored in the name field. The names of variables in intermediate code are "local_i" (for local variables), "arg_i" (for arguments), and "tmp_i" (for temporary variables). The class also contains information needed by the assembly code generator (the position it will occupy in the activation register or the processor register in which it will be stored).

  • urium.code.CodeInstruction

Class that represents an instruction in intermediate code. The kind field indicates the type of instruction (one of the codes defined in CodeConstants). The target, source1, and source2 fields contain the three addresses of the instruction. The comment field is used to generate a comment associated with the instruction, intended to describe it in a more interpretable format.

  • urium.code.CodeInstructionList

Describes a list of instructions. This class eases the generation of intermediate code by developing the concatenation of instruction lists or adding new instructions to the list.

 

 

INTERMEDIATE CODE GENERATION

 

The URIUM compiler's semantic analyzer generates a urium.ast.struct.LibraryDeclaration class that stores all recognized information in a source file written in this language. The goal of the intermediate code generator is to transform this representation into a new representation based on intermediate code instructions. The result of the generation will be a new urium.code.LibraryCodification object.

The urium.code.LibraryCodification class contains the fields name, which stores the name of the library, imported, which contains the list of imported libraries, and procedure, which stores a representation of the procedures that have already been transformed into intermediate code. In addition to the get..() methods, the class contains the print() method, which generates a description of the intermediate code to a file.

The urium.code.ProcedureCodification class describes the intermediate code associated with a procedure in a library written in URIUM. The label field stores the label that identifies the procedure and is placed as the initial label in the code. The labelcount and tmpcount fields are counters used to generate label names and temporary variables when generating the code. The arg and var fields contain references to the function's arguments and variables (local and temporary). The astvar field maintains the description of variables contained in the abstract syntax tree. This list is maintained to locate the CodeVariable object associated with each Variable object in the abstract syntax tree. Finally, the list field contains the list of instructions that the procedure executes in intermediate code. In addition to the get..() methods, the class contains the print() method, which generates a representation of the function on a file.

Intermediate code generation is performed in the private methods of the urium.code.ProcedureCodification class. The class contains several methods dedicated to generating intermediate code associated with the different structures of the abstract syntax tree. These methods include generic methods for generating code associated with a statement, an expression, and a condition. The generateCodeOfStatement() method obtains the code associated with a statement in the form of a CodeInstructionList object. The generateCodeForExpression() method stores the code associated with an expression in the codelist parameter and returns a reference to the variable that stores the result of the expression. The generateCodeForCondition() method obtains the code associated with a condition and requires the LabelTrue and LabelFalse labels to be passed as arguments to the call.

To enable the compiler to generate intermediate code files, the urium.UriumCompiler class has been modified. A new createCodeFiles() method has been added to this class, which generates the intermediate code associated with the various libraries analyzed during the compilation process.

 

 

CODE STRUCTURES

 

Intermediate code generation iterates through the instructions of each procedure (Statement objects defined in the body of the Procedure objects), generating a translation of these instructions into intermediate code (CodeInstructionList). Performing this translation sometimes requires creating new jump labels (calls to the getNewLabel() method) or new temporary variables (calls to getNewTemp()). The structure of the intermediate code generated as a translation of the different instructions, expressions, and conditions used in URIUM is described below.

  • Code structure that develops the IF-THEN statement

  • Code structure that develops the IF-THEN-ELSE statement

  • Code structure that develops the WHILE statement

  • Code structure that develops the ENDP statement

  • Code structure that develops the ASSIGNMENT statement

  • Code structure that develops a procedure call statement

    The PRECALL instruction allocates memory for the call's arguments. Each argument must generate the code associated with computing its value and a PARAM instruction. The CALL instruction contains the start label of the called procedure.

  • Code structure that develops a LITERAL expression (of type int, char or boolean)

  • Code structure that develops a BOOLEAN expression

  • Code structure that develops a unary arithmetic expression (a sign change)

  • Code structure that develops a binarty arithmetic expression

    The op instruction can be ADD, SUB, MUL, DIV or MOD.

  • Code structure that develops a reference to a variable

  • Code structure that develops a reference to address of a variable

  • Code structure that develops a condition formed by the literal TRUE

  • Code structure that develops a condition formed by the literal FALSE

  • Code structure that develops a condition formed by the negation of another condition

  • Code structure that develops a condition formed by the AND operation between two conditions

  • Code structure that develops a condition formed by the OR operation between two conditions

  • Code structure that develops a condition formed by the comparison between two expressions

    La instrucción op puede ser JMPEQ, JMPNE, JMPGT, JMPGE, JMPLT o JMPLE.

  • Code structure that develops a condition formed by a Boolean expression

 

 

EXAMPLES

 

The source code for this lecture includes the 'examples' directory with several examples of applications programmed in URIUM. The 'Compilation_scripts' subdirectory includes the compiler ('urimc.jar') and a script file ('compile.bat') with the execution command. To run the compilation, copy these two files to the application directory and run the script.

For example, the '3+2' directory contains an application written in URIUM. The contents of the 'Main.ur' file are as follows.


 import urium.Console as Console;

 /**
  * Application that shows the result of a sum
  */
 library Main {

 /**
  * Application start procedure
  */
 public proc main() {
   int a=3;
   int b=2;
   int c= a+b;

   Console.print(a);
   Console.print(' ');
   Console.print('+');
   Console.print(' ');
   Console.print(b);
   Console.print(' ');
   Console.print('=');
   Console.print(' ');
   Console.print(c);
 }
}
